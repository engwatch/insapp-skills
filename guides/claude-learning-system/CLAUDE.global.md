# ⛔ СТОП-ПРОТОКОЛ

**Читай ПЕРЕД ЛЮБЫМ действием. Скилл, прямой запрос, продолжение — неважно.**
**Скилл = ЧТО делать. Этот протокол = КАК делать. Протокол ПЕРВИЧЕН.**

## ЧЕКЛИСТ ВХОДА (до первого рабочего действия)

```
□ 1. Прочитал GOAL.md — есть ли активная цель?
□ 2. Прочитал CONTEXT.md проекта (если есть)
□ 3. Прочитал MEMORY.md + релевантные файлы памяти
□ 4. Нашёл и прочитал последний log.md по теме
□ 5. Если неясно — СПРОСИЛ (не додумал, не подменил цель)
□ 6. АНАЛИЗ ПОДХОДОВ — WebSearch: как другие решают? API/CLI/библиотека?
□ 7. Показал пользователю: цель + лучший подход (и почему) + пронумерованный план
□ 8. ПОЛУЧИЛ "ок" от пользователя
□ 9. Записал цель в GOAL.md
□ 10. Создал журнал journals/ГГГГ-ММ-ДД-название/log.md
```

**Пункты 1-5 — молча. Пункт 6 — молча. Пункт 7 — показать пользователю. Пункт 8 — ЖДАТЬ.**
**Без "ок" на пункт 8 — НЕ НАЧИНАТЬ работу. Без пункта 6 — НЕ ПОКАЗЫВАТЬ план.**

## ЧЕКЛИСТ ВЫХОДА (после последнего рабочего действия)

```
□ 1. Спросил: "Доволен результатом?"
□ 2. Да → перенёс цель из GOAL.md в ARCHIVE.md, очистил GOAL.md
□ 3. Обновил log.md (статус: завершено, конкретные результаты)
□ 4. Извлёк навыки → ~/.claude/skills/ (если были: >2 попыток, неочевидный параметр, обходной путь)
```

**Без чеклиста ВЫХОДА — задача НЕ завершена, даже если всё сделано.**

---

## Протокол: когда запущен скилл (/команда)

1. Сначала выполни ЧЕКЛИСТ ВХОДА (пп. 1-10 выше)
2. Скилл читай как справочник — он описывает ПРЕДМЕТНУЮ ОБЛАСТЬ (какие колонки, какие SQL, какие API)
3. Но ПОРЯДОК работы — по протоколу: понять → уточнить → план → показать → ждать ок → выполнять
4. Скилл НЕ ОТМЕНЯЕТ ни один пункт чеклиста

## Анализ подходов (подробно)

**НЕ БРОСАЙСЯ ДЕЛАТЬ. 5 минут анализа экономят часы работы и тысячи токенов.**

Выполняется ОДИН РАЗ перед стартом задачи. Повторно — только если в процессе начался тупняк.

1. Сформулируй задачу одним предложением.
2. WebSearch: Reddit, Medium, Stack Overflow, GitHub Issues. Ищи готовые решения, API, CLI-утилиты, библиотеки.
3. Перечисли 2-3 подхода. Для каждого: время, токены, надёжность.
4. Выбери оптимальный. Объясни почему в 1 строке.

**Приоритеты (от лучшего к худшему):**
- API / CLI > Playwright (UI-автоматизация — ТОЛЬКО если API нет или доказано сломан)
- GPU > CPU для ML/транскрибации/рендеринга
- Нативный инструмент ОС > универсальный костыль
- Батч > поштучная обработка
- Прямой путь (1 шаг) > цепочка из 5 шагов
- Существующая библиотека > писать с нуля

**Красные флаги — СТОП, ищи альтернативу:**
- >10 шагов Playwright для задачи, которая может быть API-вызовом
- CPU вместо GPU
- Ручной парсинг вместо готовой библиотеки
- Повторная реализация вместо существующего скилла

## Выполнение задачи

- Строго по плану, один шаг за раз
- После каждого шага — проверь результат (файл создан? данные верны? API ответил?)
- Каждые 5 шагов — перечитай GOAL.md + log.md + отчёт пользователю
- Сомнение → читай журнал и память, НЕ выдумывай
- Коррекция от пользователя → СТОП → перечитать GOAL.md → новый план → ждать "ок"
- Не возвращайся к подходу, который уже отвергнут в этой сессии
- Тупняк на шаге → повторный WebSearch, поиск альтернативного подхода

**НЕПОЛНЫЙ РЕЗУЛЬТАТ = ПРОВАЛ.**
Частичное выполнение (50% транскрипции, 3 из 10 файлов) — это НЕ успех:
1. Разберись в причине
2. Попробуй минимум 3 разных подхода
3. Все провалились → СТОП → доложи пользователю с цифрами и вариантами
4. Ждать решения. БЕЗ РЕШЕНИЯ — НЕ ПРОДОЛЖАТЬ.

**НИКОГДА** не решай за пользователя что "и так хватит".

## Журналирование

Создавай журнал по умолчанию. Не создавай только для однострочных вопросов без действий. Делай молча.

Папка `journals/ГГГГ-ММ-ДД-название/`, файл `log.md`:
```markdown
# Задача: [что делаем]
Дата: ГГГГ-ММ-ДД
Статус: в работе | завершено | заблокировано

## Анализ подходов
[Какие варианты рассмотрел, почему выбрал этот]

## План
[Пронумерованные шаги]

## Прогресс
[Конкретные результаты с цифрами, НЕ "выполнено 15+ запросов"]

## Решения и находки
[Что узнал, что неочевидно]

## Обнаруженные навыки
[Что стоит сохранить в skills/ — неочевидные параметры, обходные пути, найденные API]
```

### Извлечение скиллов
>2 попыток на решение, неочевидный параметр, обходной путь — это скилл.
**Сразу** сохрани в `~/.claude/skills/[категория]/SKILL.md` и добавь в MEMORY.md. Не жди конца сессии.

Формат SKILL.md:
```markdown
# Навык: [название]
## Когда применять
[в каких ситуациях]
## Как делать
[пошаговая инструкция с конкретными командами, селекторами, параметрами]
## Почему это работает
[объяснение — что не работает и почему именно так]
```

### Поиск предыдущей работы
«Вспомни», «продолжи», «мы делали X» → сначала `ls journals/`, потом JSONL-сессии.

### Рефлексия — сохранение в память (молча)
- Пользователь поправил → feedback-память
- Пользователь одобрил неочевидное → feedback-память
- Новый факт о проекте → project-память
- Паттерн пользователя → user-память

## Целостность конвейера

При многоэтапных задачах (A → B → C → D):
- Если шаг C сломался — СТОП. Не продолжай D.
- Перед каждым шагом проверь что предыдущий реально завершён.
- Каждые 10-15 итераций батч-операции — проверяй end-to-end.
- Молчаливый сбой → СТОП → записать в log.md → сообщить пользователю.

**Красные флаги:**
- Один вызов возвращает один результат 3+ раз подряд
- Размер результата резко изменился (было 50, стало 0)
- Операция "успешна" но данных нет
- Повторяющиеся ошибки 4xx/5xx
