# Report Claude Use — отчёт по использованию Claude Code

## Команда

```
/report_claude_use [период]
```

- `[период]` — опционально: "последние 2 дня", "эта неделя", "7 дней", "март". По умолчанию — **последние 3 дня**.

---

## Алгоритм

### Шаг 1. Сбор сессий

Найти все JSONL-файлы сессий в проектах:

```bash
find ~/.claude/projects/ -maxdepth 2 -name "*.jsonl" ! -path "*/subagents/*"
```

Для каждой сессии извлечь:
- **Timestamp** — из поля `timestamp` в каждой JSONL-строке (ISO 8601 или epoch ms)
- **User messages** — из записей с `type: "user"`, поле `message.content` (строка или массив с `text`)
- **Assistant texts** — из записей с `type: "assistant"`, поле `message.content[].text` (см. ниже)
- **Tools used** — из записей с `type: "assistant"`, блоки `message.content[].type == "tool_use"`, поле `name`
- **MCP tools** — инструменты начинающиеся с `mcp__`
- **Название сессии** — из записи с `type: "custom-title"`, поле `customTitle`

#### Извлечение assistant текстов (ОБЯЗАТЕЛЬНО)

**КРИТИЧЕСКИ ВАЖНО:** Для качественной суммаризации необходимо парсить **не только user messages, но и assistant texts**. Именно в ответах ассистента содержится конкретика — какие файлы изменены, какие ошибки найдены, какие решения приняты.

Из каждой записи `type: "assistant"` извлекать блоки с `type: "text"` из массива `message.content[]`:
- Брать первые **300 символов** каждого текстового блока (для экономии памяти)
- Сохранять вместе с timestamp сообщения
- Использовать при суммаризации наравне с user messages
- Исключать тексты, содержащие токены, пароли, API-ключи

#### Привязка к датам по timestamp сообщений

**ВАЖНО:** Не использовать `mtime` файла для определения даты активности. Одна сессия может содержать работу за несколько дней.

Для каждого сообщения (`user` или `assistant`) парсить поле `timestamp`:
- ISO строка: `datetime.fromisoformat(ts.replace('Z', '+00:00'))`
- Epoch ms (число > 1e12): `datetime.fromtimestamp(ts / 1000)`
- Epoch sec: `datetime.fromtimestamp(ts)`

Фильтровать: включать только сообщения, чей timestamp попадает в запрошенный период. Сессия попадает в отчёт, если хотя бы одно сообщение в ней попадает в период. В хронологии и карточках — привязывать к дате по timestamp конкретных сообщений, а не по файлу.

### Шаг 3. Суммаризация (САМЫЙ ВАЖНЫЙ ШАГ)

**ЗАПРЕЩЕНО** вставлять сырые сообщения в HTML! Каждый элемент карточки должен быть **написан тобой как резюме**.

Процесс:
1. Для каждой категории — собрать все user messages и assistant excerpts из попавших сессий
2. Прочитать их, понять контекст: что просил пользователь, что сделал ассистент
3. **Написать своими словами** 1-2 предложения на русском, что было сделано, с конкретикой (имена файлов, классов, методов, числа, результаты)
4. Привязать к дате

**Примеры правильных итогов:**
- ✅ "24.03 — Интеграция АС ДЭК v2: переработаны AlfaDecMapper, AlfaDecClient, убраны 4 промежуточных SOAP-вызова"
- ✅ "18.03 — Исправлены XML-summary в DocsPublicApi контроллерах для Swagger"
- ❌ "18.03: Пентесты" (это сырое сообщение, не резюме!)
- ❌ "— Теперь сохраню как скилл." (это сырой assistant текст!)
- ❌ "API Error: 529 overloaded_error" (это техническая ошибка, не итог работы!)

**Исключения — не включать в отчёт:**
- Сессии только с системными сообщениями или запусками скиллов без результата
- Чувствительную информацию (токены, пароли, API-ключи)
- Технические ошибки и stack traces

### Шаг 4. Генерация HTML

Структура HTML (светлые тона, Apple-стиль):

1. **Шапка** — заголовок, период, 4 KPI:
   - Количество сессий
   - Количество направлений (категорий)
   - Количество MCP-сервисов
   - Количество дней

2. **Что было сделано** — сетка 2 колонки из карточек:
   - Каждая карточка = одна категория
   - Заголовок с иконкой, названием и счётчиком
   - Список итогов с датами
   - Цветная левая граница

3. **MCP-интеграции (только использованные)** — сетка 2 колонки:
   - Показывать только MCP-сервисы, инструменты которых реально вызывались (есть `tool_use` с именем `mcp__<service>__*`)
   - Не показывать просто доступные/подключённые сервисы
   - KPI "MCP-сервисов" в шапке также считать только использованные
   - Иконка + название + краткое описание

4. **Хронология** — компактный список:
   - Группировка по дням (Пн/Вт/Ср...)
   - Сортировка по убыванию: сверху — недавние дни, снизу — давние
   - Время | категории | первое сообщение (обрезанное)

5. **Подвал** — "Claude Code (model) · дата генерации"

### Шаг 5. Сохранение и публикация

1. Сохранить HTML в `docs/claude-activity-report.html` текущего проекта
2. Открыть в браузере: `open docs/claude-activity-report.html`
3. Спросить пользователя: нужно ли опубликовать через `/html-push`
